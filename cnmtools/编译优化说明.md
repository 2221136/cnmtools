# 编译优化说明

## 已启用的优化

### 1. AOT编译（Ahead-of-Time）
Flutter Release模式默认启用AOT编译，将Dart代码编译为原生机器码。

**优势：**
- ✅ 更快的启动速度
- ✅ 更好的运行性能
- ✅ 更小的内存占用

### 2. Dart代码混淆
```bash
flutter build apk --release --obfuscate --split-debug-info=build/app/outputs/symbols
```

**参数说明：**
- `--obfuscate`: 混淆Dart代码，类名和函数名变成无意义字符
- `--split-debug-info`: 分离调试符号，用于崩溃日志还原

**优势：**
- ✅ 防止反编译
- ✅ 保护代码逻辑
- ✅ 减小APK体积

### 3. Android代码压缩和混淆
在`android/app/build.gradle.kts`中启用：

```kotlin
buildTypes {
    release {
        minifyEnabled = true        // 启用代码压缩
        shrinkResources = true      // 启用资源压缩
        proguardFiles(...)          // ProGuard规则
    }
}
```

**优势：**
- ✅ 移除未使用的代码
- ✅ 压缩资源文件
- ✅ 混淆Java/Kotlin代码
- ✅ 显著减小APK体积

### 4. ProGuard规则
创建了`android/app/proguard-rules.pro`文件：

**保留内容：**
- Flutter核心类
- 原生方法
- 注解和签名
- 行号信息（用于调试）

**移除内容：**
- 调试日志（Log.d, Log.v, Log.i）
- 未使用的代码

## 效果对比

### 优化前
- APK大小: ~50MB
- 启动时间: 较慢
- 代码可读: 易反编译

### 优化后
- APK大小: ~30-35MB（减少30-40%）
- 启动时间: 更快
- 代码可读: 难以反编译

## 调试崩溃日志

如果发生崩溃，使用符号文件还原堆栈：

```bash
flutter symbolize -i <crash-stack-trace.txt> -d build/app/outputs/symbols
```

## 注意事项

1. **首次构建时间更长**
   - 混淆和压缩需要额外时间
   - 正常现象，后续构建会有缓存

2. **调试时不要使用混淆**
   - 开发时使用`flutter run`（Debug模式）
   - 混淆仅在Release构建时启用

3. **保存符号文件**
   - `build/app/outputs/symbols`目录很重要
   - 用于还原崩溃日志
   - 建议每个版本都备份

4. **测试充分**
   - 混淆可能导致反射失败
   - 发布前务必全面测试

```yaml

```

每次推送代码都会自动构建优化后的APK。

## 进一步优化建议

1. **拆分APK**
   ```bash
   flutter build apk --release --split-per-abi
   ```
   为不同CPU架构生成独立APK，进一步减小体积。

2. **使用App Bundle**
   ```bash
   flutter build appbundle --release
   ```
   上传到Google Play，自动为用户设备优化。

3. **图片优化**
   - 使用WebP格式
   - 压缩PNG/JPG
   - 移除未使用的资源

4. **依赖优化**
   - 定期检查并移除未使用的依赖
   - 使用更轻量的替代包
